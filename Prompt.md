- доверие;
- понятную коммуникацию;
- мобильный UX;
- удобное управление контентом через админку;
- мультиязычное наполнение;
- отсутствие зависимости от WordPress и PHP.

Работай как исполнитель:

- не оставляй TODO, FIXME и placeholders;

## ОБЯЗАТЕЛЬНЫЙ СТЕК

Используй именно этот стек:

Frontend / App:

- Next.js (App Router)
- React
- TypeScript
- Tailwind CSS

Backend / CMS / Admin:

- Payload CMS, встроенный в тот же Next.js проект
- Payload admin panel
- Payload localization
- PostgreSQL

Интеграции:

- серверная обработка форм через Next.js / Payload
- SMTP для email-уведомлений
- Telegram Bot API для опциональных уведомлений

Инфраструктура:

- Docker Compose
- локальное файловое хранилище для медиа

## ОБЩАЯ ЦЕЛЬ

Нужно поддерживать и развивать современную версию сайта `Montenegro Travel`, перенесённую с WordPress/PHP на стек Next.js + Payload.

Сайт должен:

- визуально быть близок к текущему туристическому сайту;
- выглядеть аккуратно, дорого и современно;
- хорошо работать на мобильных устройствах;
- иметь удобную русскоязычную админку;
- поддерживать 4 языка контента;
- позволять редактировать контент без правки кода.

## КЛЮЧЕВОЙ UI-ПРИОРИТЕТ

Сайт не должен выглядеть как generic travel template.

Нужны:

- сильный hero;
- крупные карточки экскурсий;
- удобная фильтрация;
- аккуратная типографика;
- хорошие отступы и вертикальный ритм;
- чистый layout;
- мягкие hover/focus states;
- выразительные, но спокойные анимации;
- сильная мобильная версия.

Не нужно:

- визуальный шум;
- дешёвые карусели “ради каруселей”;
- кричащие эффекты;
- перегруженные экраны;
- ощущение “типовой CMS-темы”.

## ЭКСКУРСИИ

В проекте должна поддерживаться полноценная витрина экскурсий через коллекцию `tours`.

Каждая экскурсия должна иметь:

- локализованное название;
- категорию;
- длительность;
- сложность;
- краткое описание;
- маршрут / ключевые точки;
- примечание по цене или формату;
- обложку;
- галерею;
- порядок отображения.

Обязательные демо-направления:

- Дурмитор, мост Джуржевича, Пивское озеро
- Котор, Пераст
- Тиват, Херцег-Нови
- Будва, Петровац, Свети-Стефан
- Монастырь Острог
- Луштица
- По запросу

## РАСПИСАНИЕ

Отдельная коллекция `schedule-items`.

Каждый выезд должен поддерживать:

- связь с экскурсией;
- дату;
- время начала;
- время окончания;
- статус (`open`, `last`, `soldout`);
- локализованную точку сбора.

## ТРАНСФЕРЫ И ПАРТНЁРЫ

Ключевое бизнес-требование:

- `Трансферы` и `Партнёры` должны быть рядом в хедере;
- оба пункта должны вести в одну общую секцию;
- внутри страницы это должен быть единый визуальный блок из двух карточек рядом.

Не возвращай старую ошибку WordPress, где они визуально были рядом на странице, но в навигации и по прокрутке расходились.

## ФОРМА ЗАЯВКИ

Форма должна:

- работать на мобильных устройствах;
- не ломать премиальное ощущение интерфейса;
- валидироваться на клиенте и сервере;
- сохраняться в `inquiry-requests`;
- поддерживать honeypot;
- поддерживать rate limiting;
- опционально отправлять уведомления в email и Telegram.

Поля формы:

- имя;
- email;
- телефон;
- preferred contact method;
- тип запроса;
- интересующая экскурсия / тема;
- количество гостей;
- комментарий;
- согласие на обработку персональных данных;
- locale.

Если уведомления не настроены, форма всё равно должна корректно писать данные в БД.

Проект должен корректно работать локально:

- приложение на `3006`;

- контейнеры не должны конфликтовать с другими локальными проектами;
- использовать PostgreSQL 18;
- не тащить SQL-дампы из чужих проектов в `docker/postgres/init`.
